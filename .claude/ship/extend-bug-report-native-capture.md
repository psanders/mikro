## Decision log

Newest first. One line per meaningful decision or stage transition.

- 2026-07-04 — Manual verification (task 4.5) caught a real bug `cargo check` couldn't: `npm run tauri:dashboard:dev` crashed at launch with `dyld: Library not loaded: @rpath/libswift_Concurrency.dylib`. Root cause: `screencapturekit`'s build.rs emits the correct Swift-runtime `-Wl,-rpath,...` link args, but bare `cargo:rustc-link-arg` from a dependency's build script doesn't propagate to a downstream binary that merely depends on it — confirmed via `otool -l target/debug/mikro-dashboard` showing zero `LC_RPATH` entries. Fixed by adding the same rpath flags to `mods/dashboard/src-tauri/build.rs` itself (gated `#[cfg(target_os = "macos")]`); rebuilt, `otool -l` now shows all 3 rpaths, and the binary launches and stays running (previously crashed instantly). `cargo check` alone can't catch link/runtime issues since it skips the final link step — worth remembering for any future native-plugin work in this repo.
- 2026-07-05 — Build + automated test stages complete. Two real architecture discoveries surfaced and resolved mid-build with the user: (1) Tauri can't cheaply mux mic audio into native screen video (ScreenCaptureKit can't capture mic; muxing needs bundled/signed ffmpeg) — descoped to native screenshot + existing mic audio, full video deferred to #87; (2) `react-native-nitro-screen-recorder` has no Android in-app-only mode (Android's MediaProjection is inherently system-wide) — iOS uses `startInAppRecording`, Android uses `startGlobalRecording`, with an added Android-specific consent notice. `mods/dashboard` has zero existing frontend test infra, so Tauri correctness rests on real compilation against the actual crate rather than unit tests; mobile logic was extracted into a pure, DI'd function specifically to make it unit-testable. Manual on-device verification (real Tauri build, real EAS dev client) is NOT done — flagged before requesting the sync/archive gate.
- 2026-07-03 — Filed follow-up issue psanders/mikro#87 for attaching/linking the full video recording (not just transcript+screenshot) — deliberately kept out of this change's scope since it touches apiserver retention behavior PR #75 simplified away; user chose to keep this change scoped to client capture.
- 2026-07-03 — Design approved. Recording state redesigned as a floating non-blocking pill (red dot + timer + stop) instead of a full-screen block, after user asked whether they could move around the app while recording — correctly caught that a blocking modal would defeat multi-screen bug repro on mobile. Specs (`bug-report-mobile-capture`) and design.md updated to match (decision 4a) before final sign-off.
